# Hopper 6.2.8 Reconnaissance

Date: 2026-04-30

## Scope

This note records read-only reconnaissance of the installed Hopper bundle before the next implementation phase. It focuses on Hopper itself: bundled official MCP behavior, XPC boundaries, scripting APIs, binary capabilities, and implications for a stronger Hopper MCP.

## Installed Bundle

- App path: `/Applications/Hopper Disassembler.app`
- Bundle id: `com.cryptic-apps.hopper-web-4`
- Version: `6.2.8`
- Build: `43.6299776`
- Main executable: `Contents/MacOS/Hopper Disassembler`
- Official MCP executable: `Contents/MacOS/HopperMCPServer`
- CLI launcher: `Contents/MacOS/hopper`
- Public scripting resources: `Contents/Resources/Hopper.sdef`, `Contents/Resources/hopper_api.py`
- Relevant XPC service: `Contents/XPCServices/ExternalAPI.xpc`

All primary binaries are universal `x86_64 arm64` Mach-O files.

## Signing And Security Posture

The main Hopper app is signed by `Developer ID Application: Cryptic Apps (2AMA2753NF)`, notarized, and uses hardened runtime. Its entitlements are permissive for reverse-engineering/decompiler needs:

- `com.apple.security.cs.allow-jit`
- `com.apple.security.cs.allow-unsigned-executable-memory`
- `com.apple.security.cs.debugger`
- `com.apple.security.cs.disable-executable-page-protection`
- `com.apple.security.cs.disable-library-validation`

`HopperMCPServer` and `ExternalAPI.xpc` are signed by the same team. `ExternalAPI.xpc` has no special entitlements and is configured as a user XPC service with `JoinExistingSession = true`.

Implication: injection/private API work is feasible, but the official no-prompt automation path is already an XPC path. We should exhaust and extend that before defaulting to invasive injection.

## Official MCP Server

`HopperMCPServer` is a small signed stdio MCP process. Behavioral testing over JSON-RPC showed:

- Protocol version: `2025-03-26`
- Server info: `HopperMCPServer` `1.0.0`
- Capability: tools only, `listChanged = false`
- Transport: stdio
- It can launch or attach to Hopper and communicate with `ExternalAPI.xpc`.

The bundled Hopper Help page documents only client setup for this stdio binary. Tool behavior must be learned from live protocol calls and binary selectors.

Observed weaknesses relative to a production-grade MCP:

- Schemas are loose; most objects omit `required` and `additionalProperties: false`.
- High-cardinality tools are unpaginated and can return very large maps.
- Results are text-wrapped JSON, not structured MCP content/resources.
- Pseudocode can block for complex procedures; the official description warns about this but does not expose policy controls.
- Writes/navigation are direct live operations with no transaction preview.
- Document identity is name-based, which is ambiguous for duplicate names.

## Live Official Behavior

The official server successfully answered live calls without a manual GUI confirmation in this environment.

Observed documents:

```text
["echo", "Calculator"]
```

For the current `echo` document:

- `current_document` returned `echo`.
- `current_address` returned `0x1000004f0`.
- `search_procedures` found `EntryPoint` at `0x1000004f0`.
- `list_segments` returned Hopper-derived segments and sections.
- `procedure_info`, `procedure_assembly`, `procedure_pseudo_code`, `procedure_callees`, and `xrefs` all worked for `EntryPoint`.

Implication: we do not need GUI automation for normal live reads. The official ExternalAPI path gives the no-confirmation behavior we need, and our daemon should be able to use it as a safe backend and as a correctness oracle for private backend parity.

The XPC service imports `NSXPCConnection`, `NSXPCInterface`, and code-signing validation APIs. It appears to validate/attach to a Hopper application endpoint rather than performing analysis itself.

Implication: a better MCP can either wrap this official bridge safely or implement an additional native agent that speaks a richer local protocol. The strongest design keeps MCP in Rust and treats official/private Hopper bridges as backends.

## Public Automation APIs

### AppleScript Dictionary

`Hopper.sdef` exposes:

- `open executable`
- `open database`
- loader and CPU selection
- analysis flags
- Objective-C, Swift, and exception parsing flags
- jump/base address options
- execute Python script or command after initial analysis

### CLI Launcher

`Contents/MacOS/hopper` is `Hopper Launcher v1.6.0`. It is an AppleScript/Open-Scripting launcher, not a data extraction backend. It can open executable/database inputs and pass Python commands/scripts after analysis.

### Python API

`hopper_api.py` exposes broad public scripting classes:

- `Document`
- `Segment`
- `Section`
- `Procedure`
- `BasicBlock`
- `Instruction`
- `CallReference`
- `Tag`
- `GlobalInformation`

Important public methods include document discovery, segment/section enumeration, byte reads/writes, labels/names, comments, inline comments, xrefs, procedures, basic blocks, callers/callees, decompilation, bookmarks, colors, tags, assembly, rebasing, and executable production.

Implication: Python remains a legitimate safe backend/exporter and should not be discarded. It is slower and GUI-tied, but it provides broad public coverage and is useful as a fallback/correctness oracle.

## Third-Party Hopper MCP Landscape

Public listings for `HopperMCP` describe an Objective-C implementation that injects code into Hopper, uses reverse-engineered private interfaces, requires SIP disabling, has manual plugin/menu steps, and exposes a narrow tool set: current/by-name/by-address assembly or pseudocode plus comments.

Public listings for `HopperPy` describe a Python-based Hopper MCP/plugin approach, but available public detail is sparse compared with the official server and local Hopper resources.

The broader RE MCP landscape has much larger surfaces for other tools. Public curated lists mention Ghidra, IDA, Binary Ninja, radare2, x64dbg, Cutter, Frida, and others, with some implementations advertising dozens or hundreds of tools.

Implication: beating Hopper alternatives is achievable because existing Hopper-specific alternatives appear narrow or invasive. Beating broader RE MCPs requires not just more tools, but better correctness controls: stable identities, strict schemas, pagination, provenance, snapshots, diffing, graph search, transactions, and test fixtures.

## Design Consequences

1. `otool`/`nm`/`strings` should not be product backends. They are useful only for local corroboration and binary reconnaissance.
2. The official XPC path is the safest live authority and already avoids manual GUI confirmation for normal MCP calls.
3. A public Python exporter remains useful for broad safe coverage and regression comparison.
4. A private/native backend should be opt-in, version-gated, and fail-closed. It should extend beyond official tools with batch reads, stable IDs, richer CFG/data types, and lower latency.
5. Rust should keep owning MCP, persistence, schemas, pagination, resources, transactions, query, graph, fingerprints, diffs, and tests.
6. Objective-C++ is still the right native agent language if we need private runtime selectors or in-process/plugin behavior.
7. The next phase should target a backend abstraction around three authorities: official XPC MCP, Python exporter, and private agent/mock.

## Next Phase Recommendations

- Add a typed official-backend client that calls `HopperMCPServer` with timeouts, process cleanup, and structured parsing of text-wrapped JSON results.
- Treat official live outputs as golden cassettes for `echo`, `Calculator`, and selected larger apps.
- Add strict result normalization with provenance: backend, Hopper version/build, document identity, timestamp, caps, and truncation.
- Add pagination/caps to every high-cardinality public MCP tool.
- Build private-agent wire protocol against a mock before any injection work.
- Probe `ExternalAPIProtocol` selector compatibility at startup and expose backend diagnostics.
- Keep write operations transaction-gated even when official tools can write directly.
- Defer SIP/injection work until the safe official backend and mock private backend are fully tested.
